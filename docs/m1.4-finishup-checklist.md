## B. Cloud-sync (M2a) live checks — need real Supabase

Run in this order. These reproduce the transient-failure fault class M2a was built to survive, which
local cannot produce.

- [ ] **B1 — Round-trip.** Local-only video → sync → present in cloud; cloud-only video → sync →
      hydrated locally with a readable MD body. Confirm blob paths are `<ownerId>/<playlistKey>/<key>`
      under a **real user JWT** (Task 12 review caught a literal `{ id: 'cloud' }` principal that
      Storage RLS rejects).
- [ ] **B2 — B1 guard, live.** Make a cloud MD blob unreadable mid-sync (revoke the Storage policy
      briefly, or point at a key the policy denies). Confirm: error surfaces in `report.errors`, the
      other replica's bytes are byte-preserved, `docVersion` is not downgraded, and **no manifest
      baseline is written**. Then re-run and confirm it heals. This is the check local cannot produce.
- [ ] **B3 — serve-doc money re-run (regression already fixed).** The double-charge was confirmed and
      fixed pre-launch (`tests/integration/serve-model-unreadable.test.ts`; 6→6 spend, 0 gemini calls).
      Re-run against hosted Supabase to confirm a **real 5xx** (not simulated) carries a non-404
      `statusCode` and still does not recharge.
- [ ] **B4 — `GENERATOR_VERSION` skew (M-R7-1).** Deploy an image whose `GENERATOR_VERSION` differs
      from the local checkout; run a `copyToCloud` where both sides hold a model for the same body;
      check whether a rendering share starts returning 503.
- [ ] **B5 — No service-role on the sync path.** `npm run check:confinement`
      (`scripts/check-service-confinement.ts`) passes against the **real deployed environment**, not
      just local.
