# Claude adversarial review — Stage 1D spec (round 1)

**Date:** 2026-07-08 · Opus, fresh subagent · Target: `docs/superpowers/specs/2026-07-08-stage-1d-cost-guardrails-design.md`

## Blocking
- **B1 — The fixed 30¢ estimate is not an upper bound → the daily cap does not bound spend (defeats 1D's purpose).** Real per-summary cost is unbounded vs 30¢: `summary-handler.ts:30` allows `MAX_DURATION_SECONDS=4h`; caption-fallback `transcribeViaGemini` (`gemini.ts:501`) sends the whole video (millions of tokens); `generateSummary` re-sends the transcript up to `MAX_SUMMARY_ATTEMPTS=4` (`gemini.ts:201,281`). One long job can cost multiples of 30¢, so the ledger reserves 30¢ while real spend is far higher → `$5/day` can be blown many-fold. *Fix: derive `est` as a defensible worst-case (model price × MAX_DURATION token ceiling × attempt budget), and/or lower hosted `MAX_DURATION`; gate `est` and `MAX_DURATION` together. Do not ship a cap whose soundness rests on a flat average estimate.*

## High
- **H1 — `sweep_expired_leases` bulk release unspecified.** Sweep updates many jobs in one `UPDATE…FROM` whose CASE either requeues (must NOT release) or terminates (must release), across different enqueue-days. The single-job release snippet can't express this → mis-release (over-release → cap under-counts → overspend). Also `fail_job`'s `SELECT…FOR UPDATE` (0008:148) fetches only `attempts,max_attempts,cancel_requested` — release needs `reserved_cents,created_at`. *(Dissolved if never-release adopted.)*
- **H2 — SECURITY DEFINER owner must bypass FORCE-RLS on the new tables — unstated/untested.** `enqueue_job` (definer) must read `quota_allowance`/`guardrail_config` and write `usage_counters`/`spend_ledger`, all `force row level security`. `0006_grants.sql:9` gives BYPASSRLS only to `service_role`, not the function owner (`postgres`). If the owner lacks it, every enqueue fails closed. Precedent (`handle_new_user` definer into force-RLS `profiles`) suggests owner bypasses, but it's unstated/untested. *Fix: assert the invariant + integration-test that `enqueue_job` reads/writes all four tables; or add owner policies.*
- **H3 — Two existing schema tests change shape, not mechanically.** `tests/integration/job-queue-schema.test.ts` inserts as an authenticated session (`:18,35,47,70`): (a) "insert for another owner rejected by with-check" (`:31-40`) — after REVOKE INSERT the denial is a *grant* 42501, not the with-check policy the test names, and the spec removes with-check owner-safety; (b) "idempotency index blocks a second live job" (`:64-71`) asserts the 2nd insert errors — via `enqueue_job` it *joins* (no error), inverting the assertion. *Fix: rewrite both against `enqueue_job`'s new contract; the spec's "switch to the RPC" note undersells this.*

## Medium
- **M1 — `enqueue_job` doesn't restrict `p_job_kind`** (= Codex H4); an authenticated client can enqueue `dig` directly → phantom reserved/quota-debited unhandled rows. *Fix: reject kinds not in the enabled set.*
- **M2 — `period_start` uses session-TZ `date_trunc('month', now())`** while the daily ledger pins UTC → TZ-dependent month bucketing (allowance doubles/blocks at boundaries). *Fix: `at time zone 'utc'`.*
- **M3 — "Hard" per-IP velocity 429 is enforced only in the racy advisory preflight → not authoritative.** A concurrent burst from one IP all read the same pre-increment count and admit. Internal contradiction (spec calls preflight "advisory" yet velocity a "hard block"). *Fix: make velocity authoritative (atomic counter), or downgrade the wording; with a sound cap (B1), coarse velocity is acceptable abuse-hardening.*
- **M4 — Disjoint-sum breaks at `producer.ts:82`.** `failed = enqueueable.length - created - joined`; adding quota/cap buckets without subtracting them double-counts blocked videos into `failed`. *Fix: `failed = enqueueable - created - joined - quotaBlocked - capBlocked`.*
- **M5 — IP plumbing unspecified through the type layer** (= Codex M6). `JobQueue.enqueue`/`JobKey`/`SupabaseJobQueue.enqueue` carry no IP. *Fix: extend the signature/options.*
- **M6 — Charge-on-failure + never-refund-quota + anon allowance=2 locks anon out after two transient failures with zero output** — harsh for a "validation demo," and (in the original release-money model) asymmetric. *Fix: confirm intended; consider refunding quota on infra-terminal (dead_letter/cancelled) or raise anon allowance.*

## Low
- **L1** release doesn't zero `jobs.reserved_cents` (moot under never-release). **L2** `actual_cents` inert until deferred reconcile (acknowledged). **L3** clients can `select usage_counters` but not `quota_allowance` → UI can't show "X of N" — grant read on `quota_allowance`. **L4** notation `(now() at UTC)` vs `at time zone 'utc'` — make literal/consistent.

## Verified NOT defects (attacked, held)
`allowance=0` edge (ensure-row=0 then `update…where used<0` → quota_exceeded); daily-cap concurrency serializes on the row lock under READ COMMITTED (no overshoot, lands exactly at cap); deadlock ordering acyclic (enqueue: usage_counters→spend_ledger; fail: jobs→spend_ledger); daily-cap raise rolls back the quota debit + INSERT (one plpgsql txn); double-release once (both fence `status='active'`, sweep `for update skip locked`); REVOKE INSERT doesn't break the worker (`service_role` keeps grants + uses RPCs); charge-once (debit only in INSERT branch; auto-retry reuses the row; JOIN skips).
