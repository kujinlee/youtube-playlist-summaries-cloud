## New Findings

### High H1 - Task 12 additive publish never creates the receiver cloud video row, then can write a false baseline

Failure scenario: a local-only video is published to cloud. The presence branch chooses `to=deps.cloud` and calls `copyAdditiveVideo(to, toP, toBlob, present, body)` (`docs/superpowers/plans/2026-07-17-stage3-cloud-sync-m2a.md:2001-2008`). The helper contract says to call only `to.upsertVideo(toP, sanitizeAdditiveVideo(video))` (`docs/superpowers/plans/2026-07-17-stage3-cloud-sync-m2a.md:1954`). In the real cloud store, `upsertVideo` is not an insert/upsert; it is an `UPDATE` of a row already created by `claimVideoSlot` (`lib/storage/supabase/supabase-metadata-store.ts:88-111`). If the receiver playlist/video row does not already exist, the update affects no row and does not throw. `runSync` then writes `baselineFromOneSided` (`docs/superpowers/plans/2026-07-17-stage3-cloud-sync-m2a.md:2008-2009`), making the manifest claim the cloud has a video it never received. On the next run, local present + cloud absent + baseline present is interpreted as a remote delete and the plan suppresses recreation.

Fix: make additive create first ensure the receiver playlist and video slot exist before metadata copy. For cloud receivers, call/introduce a receiver-side create path equivalent to `resolvePlaylistId`/`setPlaylistMeta` plus `claimVideoSlot(toP, video.id)`, then upsert the sanitized record with the claimed `position`/`serialNumber` as needed. Alternatively add a `MetadataStore.createVideoIfAbsent` contract with real insert semantics and require both stores to implement it. After `copyAdditiveVideo`, verify the receiver `readIndex(toP)` contains the copied `video.id` before writing `baselineFromOneSided`. Add the assertion to the existing local-only publish test, not only T14 servability.

### Medium M1 - Task 12 additive publish can mark `artifacts.summaryMd.status='promoted'` before the MD blob exists

Failure scenario: v4 adds the requirement that additive local-to-cloud publish sets `artifacts.summaryMd.status='promoted'` (`docs/superpowers/plans/2026-07-17-stage3-cloud-sync-m2a.md:1957`), but the helper contract still orders metadata first and blob copy second: `to.upsertVideo(...)`, then `toBlob.put(...)` (`docs/superpowers/plans/2026-07-17-stage3-cloud-sync-m2a.md:1954`). If the sanitized cloud record includes promoted status as required, a crash or blob write failure after `upsertVideo` leaves a cloud row that passes the `summaryReady`/share gate (`lib/storage/supabase/supabase-metadata-store.ts:49-54`) while the MD body is missing. This is exactly the promoted-status vs sanitize interaction the round-3 fix introduced.

Fix: for additive creates with an MD body, write and verify the receiver blob before advertising promoted metadata, or reuse the same staged/promote/finalize path as `transferClassA`. Then write the baseline only after verifying both the receiver metadata row and blob. If the plan keeps direct `BlobStore.put`, reorder the helper contract to `put` + `exists/get hash verify` before `upsertVideo` with promoted artifact status.
